# Design Method Library

The knowledge base behind mentor-mode scaffolds. Organized by design stage —
the agent's lookup question is always "which tools fit the stage I'm in".
Method class (B / A / C) is an attribute governing loading, signature, and
caveats, not a file split.

Source: 八位专家对话实录 (tool × expert matrix) and its A/B classification
companion. Interaction/implementation specs from the source docs are
superseded by this skill's runtime (CanvasIR, canvas-widgets.md); what this
file keeps is the method **content**: slots, judgment rules, AI boundaries,
and attribution.

## 1. Method Classes

| 类 | 含义 | 加载规则 | Caveat |
| --- | --- | --- | --- |
| **B** | 跨领域共识方法（≥2 位专家提及），共 12 个 | 每阶段默认工具集，所有项目可用 | 无（署全部提及专家） |
| **A** | 领域特色方法（仅 1 位专家提及），目录 53 条 | 仅当对应专家在队（或作阶段客座）时推荐 | `领域特色方法，跨领域适用性未经验证` |

Signature formats: see design-experts.md §5.

## 2. Template vs Widget Judgment Rules

Hard definitions:

- **Template**（脚手架）: 静态板块化结构，引导用户把创意思维填进限定结构。每个
  板块必须有**板块名 + 一句话简介**（这个板块是什么、为什么要填）。人负责填内容；
  AI 辅助预填草稿（标注待确认）。落地为 CanvasIR section + slots + 文本节点。
- **Widget**: 交互机制承载判断的动态工具——拖拽、连线、评分、排序、倒计时、
  生成-筛选循环。人负责操作和筛选；AI 生成候选、计算聚合、实时更新。落地为
  canvas widget（见 canvas-widgets.md）。

Three judgment rules (also the default for classifying new C-class methods):

1. **关系类/网络类 → Widget**。价值在元素间的关系和权重（谁权力大、谁和谁冲突、
   哪个节点传导力强），依赖动态比较，静态表格承载不了。
2. **生成类/发散类 → Widget**。"生成→展示→筛选→再生成"的循环只有实时交互能承载。
3. **证据链类/记录类/收敛类 → Template**。价值在结构是否完整、维度是否覆盖——
   用板块+简介提醒"这里应该想到什么"。

**强制自检原则**：高频使用、低操作门槛的方法最容易被草率套用（Persona 与 HMW 被
全部 8 位专家批判性对待）。此类方法标 ⚠️，必须内置一个"使用前提检查"板块作为
强制自检项；留空或填"团队共识/推测"时触发专家警示批注（红线规则 3）。

## 3. Scaffold Conventions

- 每个方法一个根 frame；板块 = 子 slot + 简介文本；署名行 + `vd_method_source`
  meta（见 design-mentor-protocol.md §5）。
- AI 草稿内容一律标注 `AI 草稿，待确认`，与用户内容视觉区分。
- 遵守各方法的 AI 角色边界（下文每个方法注明）。
- 方法链（推荐数据流，非强制）：

```text
Persona → 用户旅程 → 利益相关者地图 → 利益冲突映射 → HMW
  → AI 方案发散 → 低精度原型 → 共创工作坊 → 设计导则 / 服务蓝图
```

下游脚手架创建时应引用上游产出（如冲突映射读取利益地图的角色清单），并用
connector 标注数据关系。

## 4. Discover 发现（发散）

### B 类规格

#### Persona 用户画像卡 — Template ⚠️

- 署名：马谨、刘洋、魏佛兰、吴端、孙效华、娄永琪、辛向阳、王受之（全员批判性认可）
- 用途：构建典型用户画像。**入门级但风险最高的工具**——画像必须从数据里长出来，
  不能凭假设画。
- 板块：基本信息（给这个人一个具体锚点）｜情境画像（扎根具体情境，不是悬空特征
  清单）｜目标与动机（区分"说出来的需求"和"真正的动机"）｜痛点与障碍（每条标注
  "观察到的"还是"团队推测的"）｜行为习惯（记录"做什么"而非"怎么说自己"）｜
  一句话引言（用这个人自己的语气，避免设计师替他说话）｜**数据来源标注 ⚠️**
  （强制自检：基于真实访谈聚类还是假设草拟）
- AI 角色：可从访谈笔记/聚类结果草拟画像，标注"AI 推测，需验证"。
- 链接：下游 → 用户旅程地图、HMW。

#### 用户旅程地图 — Template

- 署名：刘洋、吴端、孙效华、辛向阳
- 用途：可视化完整路径与触点。旅程是跟着真实用户走出来的，不是设计出来的；
  人机场景下还需标注控制权转移（孙效华）。
- 板块：阶段划分（从真实观察提炼，不套通用模板）｜触点（物理+数字都记）｜用户行为
  （客观动作，不猜意图）｜想法与感受（要有观察依据）｜痛点｜机会点（标"初步想法"）｜
  情绪曲线（-5~5，定位关键转折点）
- AI 角色：可根据 Persona 与观察笔记预填阶段骨架草稿。
- 链接：上游 ← Persona；下游 → 利益相关者地图、服务蓝图。

#### 利益相关者框架地图 — Widget

- 署名：马谨、娄永琪、王受之
- 用途：画权力结构，不只画"谁是谁"——谁有权力定义问题、谁的声音被忽略。
- 数据：角色[{名称, 权力等级, 声音被听见程度}]，关系[{A, B, 类型: 合作/冲突/依赖}]
- 交互要点：节点图（大小=权力，深浅=声量），可拖拽、拉线标注关系；支持"关系视图 ↔
  权力-声量四象限散点"切换。v1 运行时可先用四象限散点降级实现力导向图。
- AI 角色：从项目描述预判候选角色；自动提示"权力高但常被单独访谈"与"权力低但易被
  会议忽略"两类风险，人工确认。
- 链接：下游 → 利益冲突映射（读取角色清单）。

#### 浸泡式田野感知（田野日志） — Template

- 署名：魏佛兰、娄永琪、辛向阳
- 用途：操作化"浸泡"——不带方案进场，先用身体感知。带着方案进场的那一刻，场域就
  变成了项目用地。
- 板块：日期/时长/地点（最低限度结构化）｜无结构观察记录（自由文本，不设引导问题，
  防止把浸泡异化成任务清单）｜意外发现（唯一强制追问，捕捉计划外观察）｜身体/情绪
  反应（记录自己的感受，不只记"客观事实"）。每次记录一个新 frame，以日期命名。
- AI 角色：**克制**——不主动生成建议或补全；仅在多天记录积累后做一次跨日主题聚合。

## 5. Define 定义（收敛）

### B 类规格

#### HMW 问题卡 — Template ⚠️

- 署名：全员 8 位（唯一"全员批判性认可"的方法）
- 用途：把模糊挑战转化为"我们如何能……"。**收尾工具而非起点工具**：前置检查是
  刚需——先做层级定位/冲突映射，再写 HMW。
- 板块：观察到的现象（来自 Discover 的具体现象，不是抽象问题）｜受影响人群（具体
  到角色）｜期望的改变（描述"变成什么样"，不是"用什么方案"）｜HMW 陈述（模板句式
  生成，可微调；人机项目标注主体是人/系统/协作——孙效华）｜**使用前提检查 ⚠️**
  （强制自检：从共识讨论来的，还是看见真实分歧后写的——共识产物大概率是假问题）
- AI 角色：基于 Discover 聚类结果推荐候选 HMW 供选择/改写。
- 链接：上游 ← 利益冲突映射；下游 → AI 方案发散。

#### 利益冲突映射 — Widget

- 署名：娄永琪、辛向阳
- 用途：利益相关者地图的深化层——标注谁和谁的利益在打架、谁的权力被忽略；对应
  交互五要素的"目的"维度冲突。
- 数据：{角色对[A,B], 冲突点描述, 冲突强度 1-5, 是否可调和}
- 交互要点：在利益地图上叠加冲突连线（粗细=强度，黄=可调和/红=不可调和）；点击
  连线展开详情；冲突强度排序视图；导出冲突清单为讨论议题。
- AI 角色：根据各角色诉求自动检测两两潜在冲突、生成候选连线与初评强度，用户裁决。

#### 针灸位点识别器 — Widget

- 署名：魏佛兰、娄永琪
- 用途：找"动了它周围就开始自己调整"的传导力最强节点，不是问题最大的地方——
  决定优先干预顺序。
- 数据：{系统节点[{名称, 问题严重度 1-10, 传导力 1-10}]}
- 交互要点：二维散点（横轴=严重度，纵轴=传导力——**两维必须分开呈现**）；右上
  象限自动高亮"优先干预候选"；节点点开"动这里会带来哪些连锁变化"推演框；
  按严重度/传导力/综合分切换排序。
- AI 角色：根据观察记录与关系图连接数给初步传导力评分建议，用户校正。

## 6. Develop 发展（发散）

### B 类规格

#### 低精度原型工具包 — Template

- 署名：马谨、孙效华、娄永琪、王受之
- 用途："用原型提问，不是用原型证明你对了。"低精度让人敢批评、聚焦核心逻辑；
  每版原型回头对照资料库检验（王受之）。
- 板块：待验证假设（原型用来"检验"不是"展示"）｜保真度等级（纸面/线框/可点击——
  刻意限定选项防止过早精细化）｜原型描述/媒介（手绘、卡片、拼贴皆可，可传图）｜
  测试记录（标注这条反馈是否被认真提出批评）｜迭代方向。每版新增 frame，标注
  时间与版本号，保留历史。
- AI 角色：基于待验证假设建议该轮应聚焦测试的交互点。
- 链接：上游 ← AI 方案发散；下游 → 共创工作坊、导则。

#### 参与式共创工作坊记录表 — Template

- 署名：马谨、魏佛兰、吴端
- 用途：让利益相关者参与方案**生成**（而非仅验证），并强制反思权力不对称——
  假装平等反而强化不平等。
- 板块：参与者名单与角色｜使用材料（记录动手媒介，不只是讨论）｜产出方案（描述+
  照片）｜**权力不对称观察 ⚠️**（"谁的声音没被充分表达"）。每场工作坊一个 frame。
- AI 角色：根据发言/贡献分布粗略提示是否集中在少数人身上。

#### AI 辅助方案发散器 — Widget

- 署名：刘洋、孙效华、王受之
- 用途：AI 在给定方向快速生成大量变体，帮设计师看到更多可能性。**核心原则：AI 只
  扩大方案空间，不替代判断**——界面不设"AI 推荐首选"标记。
- 数据：{给定方向, AI 生成变体[{方案描述, 生成逻辑说明}], 人工筛选状态: 保留/淘汰/待定}
- 交互要点：卡片流展示多条变体，每张标注生成逻辑（"从功能创新角度"）；卡片拖入
  保留/淘汰/待定三分区；"重新生成本方向""换一个方向"按钮；导出保留区为下一阶段
  输入清单。
- AI 角色：生成变体与逻辑说明；不排序、不推荐。

## 7. Deliver 交付（收敛）

### B 类规格

#### 设计导则与原型库 — Template

- 署名：吴端、孙效华、王受之
- 用途：不只交付方案，同时交付可复用的设计模式，让后来者站在你的肩膀上。

## 9. Generation Model — Specs Become Canvas Objects On Demand

本库的每个条目是**规格（spec），不是预制件**。智能体按项目需求现场生成：先用
§2 判定规律确定该做 Template 还是 Widget，再从规格生成贴合当前项目的画布对象。

- **Template 类**：按规格的板块表用 CanvasIR 生成（section + slot 简介文本 +
  `add_node` 组合）。板块措辞、示例、AI 预填草稿都要贴合当前项目的领域语言和
  语境，不套通用文案。
- **Widget 类**：规格中的"数据结构 + 交互要点 + AI 角色边界"是**生成简报**，
  走 widget 生成管线（canvas-widgets.md：运行时组装器 + `window.vd` 状态桥 +
  校验修复阶梯）生成自由 HTML 片段。方法本身**不做成预制 widget 模板**——
  同一个"针灸位点识别器"在不同项目里应该长得不一样。内置的通用交互原语
  （投票、对齐量表、评分、条形图、词云、计时器）可在恰好匹配时参数化调用，
  或作为生成时的交互参考。
- **实例沉淀**：生成得好的实例可归档进项目脚手架库（`POST /api/scaffolds`）
  供本项目复用——沉淀的是项目化的实例，不是硬编码的通用模板。
- ⚠️ 强制自检警示 = 专家署名的红色 annotation（design-mentor-protocol.md §5）。
