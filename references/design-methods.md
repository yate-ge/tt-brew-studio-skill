# 设计方法库

这是导师模式脚手架背后的知识库，按设计阶段组织。agent 查询时先问两个问题：
“这轮场景是什么？”以及“当前阶段适合哪些工具？” 方法类别（B / A / C）只决定加载规则、
署名和 caveat，不代表文件拆分。

本文是 agent 参考库，不是学生可见的方法菜单、工具栏、模板库或固定 Widget 库。实际使用
路径是：agent 阅读方法规格，结合当前阶段、专家组、画板证据和用户目标，决定生成一个项目化
的视觉脚手架（方法模板、交互组件，或二者组合）。文中的 Widget 例子用于说明可能的交互
形态，不是可用组件的穷尽清单；具体 Widget 由智能体按项目语境自行决定、生成或组合。

方法模板的判断契约（何时用、6 个结构母型、三层生成策略、三段生命周期、结构自检、专家角色）
见 [canvas-templates.md](canvas-templates.md)；详细示例见 [template-examples.md](template-examples.md)。
本文提供每个方法的具体规格（板块、署名、AI 角色）。

来源：八位专家对话实录（tool × expert matrix）及其 A/B 分类配套文档。源文档中的交互和
实现规格由本 skill 运行时（CanvasIR、canvas-widgets.md）接管；本文保留的是方法**内容**：
slots、判断规则、AI 边界和归因。

## 0. 场景先于方法

同一个阶段会因为场景不同而选择不同脚手架。不要只看阶段名就套模板；先判断本轮动作：

| 场景 | 方法选择重点 | 常见输出 |
| --- | --- | --- |
| 项目导入 | 低门槛起步、建立证据入口和专家契约 | brief 头卡、专家组、当前阶段 starter scaffold |
| 继续推进 | 让学生完成下一步设计动作，而不是追加说明 | 当前阶段方法模板、上游证据 connector、下一步动作区 |
| 局部生成 | 用控制台或方法模板限定生成范围，并标注 AI 草稿 | artifact 草案、版本链、生成参数、专家评审 |
| 反馈回应 | 围绕被批注对象建立修订或风险脚手架 | 专家回复、修订任务卡、证据补充 slot |
| Widget 输出处理 | 把交互结果转成长期可编辑知识 | 决策记录、Persona/HMW/方案卡、阶段小结 |
| 阶段转换 | 收束旧阶段，打开新阶段 | 阶段小结、搁置清单、下一阶段 starter scaffold |
| 全项目诊断 | 四阶段全部承接证据，再找跨阶段张力 | 四阶段诊断包、跨阶段针灸式诊断、专家意见 |
| 局部评审 | 对目标对象做结构化批评 | rubric、局部诊断卡、修订建议 |

场景决定覆盖范围；阶段决定具体方法；专家组决定 A 类方法是否可用。所有场景都遵守：
先创建视觉脚手架，再填内容；Widget 稳定结果要物化；专家意见绑定具体对象。

## 1. 方法类别

| 类 | 含义 | 加载规则 | Caveat |
| --- | --- | --- | --- |
| **B** | 跨领域共识方法（≥2 位专家提及），共 12 个 | 每阶段默认工具集，所有项目可用 | 无（署全部提及专家） |
| **A** | 领域特色方法（仅 1 位专家提及），目录 53 条 | 仅当对应专家在队（或作阶段客座）时推荐 | `领域特色方法，跨领域适用性未经验证` |
| **C** | 即席合成方法（目录之外，按需生成） | 按 §2 判定规律 + §3 脚手架规范现场合成 | `即席方法，用后需评估`；用后可归档进项目脚手架库 |

署名格式见 design-experts.md 第 5 节。

框架来源视角（与 design-mentor-protocol.md §4.1 生成管线对应）：**甲｜既有方法框架**
= 本目录的 A/B 类方法；**乙｜项目本质框架** = C 类的主要形态——专家把自己领域的概念 /
设计 / 交互框架针对当前项目本质现场构建（例：为 agent skill 项目构建"组成 / 流程 / 功能"
框架，为 AI 产品构建 human-agent interaction 框架）。乙类框架同样遵守 §3 脚手架约定：
根 frame、板块 + 简介、一句话说明和 `vd_method_source`。

## 2. 方法模板 vs 交互组件判断规则

硬定义：

- **方法模板（CanvasIR Template）**：静态板块化脚手架，引导学生把创意思维填进限定
  结构。每个板块必须有**板块名 + 一句话简介**（这个板块是什么、为什么要填）。学生负责
  填内容；AI 只能辅助预填草稿（标注待确认）、提示风险或提出追问。运行时落地为 CanvasIR
  section、slot、text、sticky、shape、connector 等原生画板节点。
- **交互组件（Widget）**：用交互机制承载设计动作的状态化微型工具，例如输入、提交请求、
  拖拽、连线、评分、排序、倒计时、生成-筛选循环、结果可视化、参数模拟。学生负责操作、
  校正和筛选；智能体可以生成候选、计算聚合、处理异步请求并回写结果。运行时落地为 canvas
  widget / `html_component`（见 canvas-widgets.md）。

术语关系：

```text
静态设计方法脚手架 = 方法模板 = CanvasIR Template
动态交互判断工具 = 交互组件 = Widget
```

不要把“方法模板”和“交互组件模板”混在一起：`/api/canvas-templates` 暴露的是可复用的
CanvasIR 方法模板；`/api/canvas-widget-templates` 暴露的是交互组件的参数化模板。

三条判断规则（也是新 C 类方法的默认分类规则）：

1. **证据链 / 记录 / 收敛类 → 方法模板**。价值在结构是否完整、维度是否覆盖、证据是否
   可追踪；用板块 + 简介提醒“这里应该想到什么”。
2. **关系类 / 网络类 → 优先交互组件**。价值在元素间关系、权重、冲突、传导力和动态比较；
   如果只是静态描摹关系，可用方法模板；如果需要拖动、连线、权重调整、排序或视图切换，
   用交互组件。
3. **生成 / 发散 / 筛选循环 → 优先交互组件**。“生成 → 展示 → 筛选 → 再生成”的循环只有
   实时交互能充分承载。若只是记录一次发散结果，则用方法模板。
4. **默认保守规则**：能用原生画板节点和方法模板清楚表达的，不要做成 Widget。Widget 应
   用在交互确实训练某种判断力时，而不是为了显得高级。

**强制自检原则**：高频使用、低操作门槛的方法最容易被草率套用（Persona 与 HMW 被
知识库中的八位来源专家共同批判性对待，但同场在场专家仍不超过 5 位）。此类方法标 ⚠️，必须内置一个"使用前提检查"板块作为
强制自检项；留空或填"团队共识/推测"时触发专家警示批注（红线规则 3）。

## 3. 脚手架约定

- 每个方法一个根 frame；板块 = 子 slot + 简介文本；专家归属只写入 `vd_method_source`
  meta（见 design-mentor-protocol.md §5），不在画布上放署名行。
- AI 草稿内容用浅色样式与用户内容视觉区分，并在 meta/语义层标记待确认。
- 遵守各方法的 AI 角色边界（下文每个方法注明）。
- 每个方法实例都必须显式写出它训练的设计判断力、学生亲手要完成的动作、专家的审美
  介入点，以及 AI 不可代替学生完成的部分。
- 方法链（推荐数据流，非强制）：

```text
Persona → 用户旅程 → 利益相关者地图 → 利益冲突映射 → HMW
  → AI 方案发散 → 低精度原型 → 共创工作坊 → 设计导则 / 服务蓝图
```

下游脚手架创建时应引用上游产出（如冲突映射读取利益地图的角色清单），并用
connector 标注数据关系。

### 脚手架说明要求

每个生成到画板上的方法模板或交互组件，顶部只放**一句话说明**（不是 7 字段头卡）：

```text
{大标题：方法名}            text_size: xl, font: sans
{一句灰色小字：你要做什么}   text_size: s, color: grey
```

- 说清是哪种模式：**空脚手架**（"请你观察/思考/填入……"）或 **AI 预填**（"我先草拟了几条，
  请你确认、修改或讨论"）。
- "训练什么判断 / 可能遮蔽 / AI 边界 / 专家署名"这些**不上画布**——它们是 agent 的思路和机器
  meta（`vd_method_source`→左侧专家坝），不是画布内容。完整规范见
  [canvas-templates.md](canvas-templates.md) §6。

每个分区再配一句引导问题（`text_size: s, color: grey`），学生就知道这格该想什么、该填什么，
而不是面对一个空格子。

## 4. Discover 发现（发散）

### B 类规格

#### Persona 用户画像卡 — 方法模板 ⚠️

- 署名：马谨、刘洋、魏佛兰、吴端、孙效华、娄永琪、辛向阳、王受之（全员批判性认可）
- 用途：构建典型用户画像。**入门级但风险最高的工具**——画像必须从数据里长出来，
  不能凭假设画。
- 板块：基本信息（给这个人一个具体锚点）｜情境画像（扎根具体情境，不是悬空特征
  清单）｜目标与动机（区分"说出来的需求"和"真正的动机"）｜痛点与障碍（每条标注
  "观察到的"还是"团队推测的"）｜行为习惯（记录"做什么"而非"怎么说自己"）｜
  一句话引言（用这个人自己的语气，避免设计师替他说话）｜**数据来源标注 ⚠️**
  （强制自检：基于真实访谈聚类还是假设草拟）
- AI 角色：可从访谈笔记/聚类结果草拟画像，标注"AI 推测，需验证"。
- 链接：下游 → 用户旅程地图、HMW。

#### 用户旅程地图 — 方法模板

- 署名：刘洋、吴端、孙效华、辛向阳
- 用途：可视化完整路径与触点。旅程是跟着真实用户走出来的，不是设计出来的；
  人机场景下还需标注控制权转移（孙效华）。
- 板块：阶段划分（从真实观察提炼，不套通用模板）｜触点（物理+数字都记）｜用户行为
  （客观动作，不猜意图）｜想法与感受（要有观察依据）｜痛点｜机会点（标"初步想法"）｜
  情绪曲线（-5~5，定位关键转折点）
- AI 角色：可根据 Persona 与观察笔记预填阶段骨架草稿。
- 链接：上游 ← Persona；下游 → 利益相关者地图、服务蓝图。

#### 利益相关者框架地图 — 交互组件

- 署名：马谨、娄永琪、王受之
- 用途：画权力结构，不只画"谁是谁"——谁有权力定义问题、谁的声音被忽略。
- 数据：角色[{名称, 权力等级, 声音被听见程度}]，关系[{A, B, 类型: 合作/冲突/依赖}]
- 交互要点：节点图（大小=权力，深浅=声量），可拖拽、拉线标注关系；支持"关系视图 ↔
  权力-声量四象限散点"切换。v1 运行时可先用四象限散点降级实现力导向图。
- AI 角色：从项目描述预判候选角色；自动提示"权力高但常被单独访谈"与"权力低但易被
  会议忽略"两类风险，人工确认。
- 链接：下游 → 利益冲突映射（读取角色清单）。

#### 浸泡式田野感知（田野日志） — 方法模板

- 署名：魏佛兰、娄永琪、辛向阳
- 用途：操作化"浸泡"——不带方案进场，先用身体感知。带着方案进场的那一刻，场域就
  变成了项目用地。
- 板块：日期/时长/地点（最低限度结构化）｜无结构观察记录（自由文本，不设引导问题，
  防止把浸泡异化成任务清单）｜意外发现（唯一强制追问，捕捉计划外观察）｜身体/情绪
  反应（记录自己的感受，不只记"客观事实"）。每次记录一个新 frame，以日期命名。
- AI 角色：**克制**——不主动生成建议或补全；仅在多天记录积累后做一次跨日主题聚合。

## 5. Define 定义（收敛）

### B 类规格

#### HMW 问题卡 — 方法模板 ⚠️

- 方法来源：八位来源专家共同批判性认可；画布归属按当前在场专家组写入 `vd_method_source`
- 用途：把模糊挑战转化为"我们如何能……"。**收尾工具而非起点工具**：前置检查是
  刚需——先做层级定位/冲突映射，再写 HMW。
- 板块：观察到的现象（来自 Discover 的具体现象，不是抽象问题）｜受影响人群（具体
  到角色）｜期望的改变（描述"变成什么样"，不是"用什么方案"）｜HMW 陈述（模板句式
  生成，可微调；人机项目标注主体是人/系统/协作——孙效华）｜**使用前提检查 ⚠️**
  （强制自检：从共识讨论来的，还是看见真实分歧后写的——共识产物大概率是假问题）
- AI 角色：基于 Discover 聚类结果推荐候选 HMW 供选择/改写。
- 链接：上游 ← 利益冲突映射；下游 → AI 方案发散。

#### 利益冲突映射 — 交互组件

- 署名：娄永琪、辛向阳
- 用途：利益相关者地图的深化层——标注谁和谁的利益在打架、谁的权力被忽略；对应
  交互五要素的"目的"维度冲突。
- 数据：{角色对[A,B], 冲突点描述, 冲突强度 1-5, 是否可调和}
- 交互要点：在利益地图上叠加冲突连线（粗细=强度，黄=可调和/红=不可调和）；点击
  连线展开详情；冲突强度排序视图；导出冲突清单为讨论议题。
- AI 角色：根据各角色诉求自动检测两两潜在冲突、生成候选连线与初评强度，用户裁决。

#### 针灸位点识别器 — 交互组件

- 署名：魏佛兰、娄永琪
- 用途：找"动了它周围就开始自己调整"的传导力最强节点，不是问题最大的地方——
  决定优先干预顺序。
- 数据：{系统节点[{名称, 问题严重度 1-10, 传导力 1-10}]}
- 交互要点：二维散点（横轴=严重度，纵轴=传导力——**两维必须分开呈现**）；右上
  象限自动高亮"优先干预候选"；节点点开"动这里会带来哪些连锁变化"推演框；
  按严重度/传导力/综合分切换排序。
- AI 角色：根据观察记录与关系图连接数给初步传导力评分建议，用户校正。

### 四阶段项目诊断包 — 方法模板组 + 可选交互组件（诊断再入场专用）

- 署名：娄永琪（针灸式诊断 / 系统传导力）、王受之（先占资料 / 资料中长出框架）、
  孙效华（交互控制权与真实使用场景）；具体项目再叠加主导专家或虚拟专家。类别按 B 类对待，
  因为"看着已有项目找张力"不绑定单一领域。
- 触发：全项目诊断、评审、模糊优化请求（"帮我诊断这个项目 / 重新评审整个项目 / 帮我优化这个
  项目 / 哪里可以改进"），见 design-mentor-protocol.md §4.1。若用户指定单一对象，则改用局部
  阶段诊断。
- 用途：接住模糊的"优化"，把一个已存在项目按 Discover / Define / Develop / Deliver 四个
  视角拆开，使资料、定位、方案和交付都成为可批评证据，再找传导力最强的设计杠杆点。它替代
  "一上来就生成优化后成品"这个反模式。
- 前置：必须已有专家团队和四阶段导师工作区；缺任一项时先补齐。全项目诊断不得只落入一个
  当前阶段 frame。

#### 四阶段诊断结构

1. **Discover：资料与前期发散盘点 — 方法模板为主，可配可视化 Widget**
   - 推荐母型：分区收集型 / 关系网络型 / 序列链型。
   - 最小板块：已有资料（文档、代码、图片、访谈、参考）｜前期想法｜调研/观察结果｜
     灵感来源｜证据强度｜待补资料。
   - 可选 Widget：词云、topic cluster、证据强度条、资料时间线。只有当 agent 已从项目材料
     提取出真实词频、主题、时间或来源时才用。
   - AI 边界：可从项目文件和画板内容提取资料索引，标 `AI 草稿，待确认`；不能把未读材料伪造成
     证据。没有资料时写 `未发现 / 待补`。

2. **Define：定位、用户、场景与问题定义 — 方法模板为主，必要时加判断 Widget**
   - 推荐母型：卡片字段型 / 序列链型 / 网格矩阵型。
   - 最小板块：产品/服务定位｜目标用户或利益相关者｜关键场景｜现有问题陈述｜
     HMW 或设计张力｜评价标准 ⚠️。
   - 可选 Widget：HMW 检查、定位清晰度量表、Persona 证据核验、优先级矩阵。
   - AI 边界：可根据证据草拟 Persona、场景和 HMW，必须标注证据来源或推测；评价标准留空、
     填"团队共识 / 猜测"或只写"更好看"时，由方法论锚点发 ⚠️ 警示。

3. **Develop：方案结构、信息架构、功能展开与迭代 — 模板和 Widget 组合**
   - 推荐母型：序列链型 / 分区收集型 / 关系网络型 / 时序层带型。
   - 最小板块：方案或功能清单｜信息架构 / 流程｜版本链或迭代时间线｜关键设计决策｜
     已知问题与取舍｜下一轮验证。
   - 可选 Widget：方案筛选、迭代时间线、关系网络、优先级矩阵、参数模拟器。Widget 中的节点
     必须来自真实方案、文件、页面、组件或历史记录。
   - AI 边界：可整理已有方案、推断信息架构、标出冲突和缺口；不能替学生直接裁定哪个方案最终
     胜出。

4. **Deliver：可交付产物、架构、原则、导则与风险 — 方法模板为主**
   - 推荐母型：卡片字段型 / 序列链型 / 分区收集型。
   - 最小板块：当前可交付产物 / UI / artifact｜系统或内容架构｜设计原则｜使用/验收导则｜
     风险与搁置清单｜交付准备。
   - 可选 Widget：最终评审 rubric、交付准备 tracker、风险热力图、决策链追踪。
   - AI 边界：可归纳模式、风险和导则草稿；不能把草稿伪装成已确认交付标准。

5. **跨阶段：针灸式诊断 — 只有四阶段证据落位后才创建**
   - 推荐形式：交互组件（二维散点：横轴=问题严重度，纵轴=传导力）或可编辑关系图。
   - 数据要求：每个点必须引用来源阶段、来源 slot / shape、证据摘要和待验证假设。
   - 学生动作：校正严重度与传导力 → 选择 1-3 个想优先推进的杠杆点 → 收窄成一句设计张力。
   - AI 边界：可给初评建议，不排序、不推荐首选；选择权在学生手里。

#### 最小画板产物

全项目诊断最少生成四个 `scaffold.root`（每阶段一个）和一个可选跨阶段诊断 root。每个阶段 root
必须包含：

- 顶部一句话说明（大标题 + 灰色学生动作）和 `vd_method_source` 专家来源 meta。
- 3-6 个针对项目的 slot，而不是通用空格子。
- 从项目证据中提取的 seed 内容，或清楚的 `未发现 / 待补` 标记。
- 一个"学生下一步动作"区。
- 至少一个可被专家意见绑定的关键对象（root、slot、Widget 或 connector）。

专家诊断最小输出：每阶段至少一条专家意见绑定到具体对象，另有一条跨阶段意见命名主要设计张力。
主导专家看整体张力；方法论锚点检查证据链、问题定义和评价标准；领域专家或虚拟专家补充语境风险。

#### 训练的判断力

学生练习从资料中长出框架，而不是从模板倒灌内容；区分"资料不足"、"问题未定义"、"方案结构混乱"、
"交付标准不清"四种不同问题；最后再判断哪个点最有传导力。

#### 可能遮蔽

四阶段框架会让一些非线性回跳显得像流水线；因此要用 connector 和专家意见标出阶段之间的回流
关系。针灸散点会低估尚无证据但影响深远的结构性问题；用 Discover 的待补资料区和专家警示补位。

#### 反模式

- 一个全项目诊断只生成一个 Develop 阶段诊断包。
- 先写诊断结论，再补模板。
- Widget 没有真实数据，只放抽象"杠杆点"。
- 四阶段中某个阶段完全空白。
- 专家意见不绑定具体脚手架对象。

## 9. 生成模型：规格按需变成画板对象

本库的每个条目是**规格（spec），不是预制件**。智能体按项目需求现场生成：先用
§2 判定规律确定该做方法模板（CanvasIR Template）还是交互组件（Widget），再从规格生成
贴合当前项目的画板对象。

- **方法模板类**：按规格的板块表用 CanvasIR 生成（section + slot 简介文本 +
  `insert_template` 或 `add_node` 组合）。板块措辞、示例、AI 预填草稿都要贴合当前项目的
  领域语言和语境，不套通用文案。必须加入 §3 的一句话说明，并把 `学生动作` 设计成清楚可
  执行的填写、观察、比较或测试任务。
- **交互组件类**：规格中的"数据结构 + 交互要点 + AI 角色边界"是**生成简报**，
  走 widget 生成管线（canvas-widgets.md：运行时组装器 + `window.vd` 状态桥 +
  校验修复阶梯）生成自由 HTML 片段。方法本身**不做成预制 widget 模板**——
  同一个"针灸位点识别器"在不同项目里应该长得不一样。内置的通用交互原语
  （投票、对齐量表、评分、条形图、词云、计时器）可在恰好匹配时参数化调用，
  或作为生成时的交互参考。交互组件必须把 AI 产生的候选、计算或聚合与学生的选择、
  校正、淘汰、搁置动作分开呈现。
- **实例沉淀**：生成得好的实例可归档进项目脚手架库（`POST /api/scaffolds`）
  供本项目复用——沉淀的是项目化的实例，不是硬编码的通用模板。
- ⚠️ 强制自检警示 = 专家署名的红色 annotation（design-mentor-protocol.md §5）。
