# 协同模型：专家如何在画布上加内容、批注与共创

本文定稿"专家像真人一样在共享画布上协作"的交互模型，配合权威 case
[case-playable-city.md](case-playable-city.md)。原则：**专家添加的东西直接融进画布（不堆署名
文字），但一切都连回对应专家**（`meta.vd_method_source` → 左侧专家坝 + 连线）。

## 1. 专家如何"添加内容"（直接加，都连到专家）

专家不只搭脚手架，也像真人一样**直接往画布加东西**，按语义选原生工具：

- **Frame / 方法模板**：专家生成的思考框架（叙事 1）。哪位专家加的 Frame → 连到那位专家。
- **便签 `note`**：专家可以直接贴便签补一个想法 / 示例 / 提示 —— 直接落在画布上，连到该专家。
- **形状 / 连线 / 图片**：图示、关系、参考材料。

**共同规则**：每个专家添加物在 `meta.vd_method_source`（或等价作者 meta）里记归属，运行时让
**左侧专家坝**亮起并画连线到该对象。**画布上不写署名文字**；想知道"这来自谁"，看左侧专家、
悬停连线。

**标注工具（紫色区域矩形 / 紫色箭头 / 对象批注）是额外的、用户专属**：只有小雷用来表达反馈，
agent 永不用它们作图。

## 2. Frame 头像与使用说明（叙事 1）

每个专家 Frame **左上角带对应专家头像**；**鼠标悬停显示该专家对这个 Frame 的使用说明**
（"这个 Frame 帮你看什么、怎么用"）。数据来自 `meta.vd_method_source.experts` 和
`meta.vd_usage_note`。

## 3. 专家批注 / comment（叙事 3，产品亮点）

专家评论以**画布上的彩色贴纸**呈现（不是只在左侧栏）：

- **贴纸本身颜色 = 评论类型**：🔴 红 = 重大调整 · 🟡 黄 = 小修建议 · 🟣 紫 = 补充参考 / 方法。
  用**高级感**配色（克制、柔和的高级色调，不要刺眼的原色）。
- 每张贴纸**有箭头指向具体内容**，**右下角带专家头像**。
- **点击专家头像 → 只高亮该专家的所有意见**（其它淡出）。
- 批注正文只写专家真正想说的话；`观察/依据/遮蔽/下一步` 等思路进 `meta`，不作为字段标签显示。

> 与"署名不上画布"不矛盾：脚手架**作者归属**不写文字（只连专家坝）；但专家**评论**是协作动作，
> 要在画布上可见（彩色贴纸 + 头像 + 箭头），并同样连回专家坝。

### 3.1 两套颜色，不能混（重要）

- **① 贴纸本身颜色**：按评论**类型**（红/黄/紫，见上），是贴纸的填充/主色。
- **② 外围 highlight / 关联描边**：hover / 选中时用来标示关联的**描边**，与①**完全独立**：
  **专家关联 = 黄色**，**用户目标 = 紫色**。
  - 一张紫色贴纸（参考型）也可能有黄色 highlight（因为它是某位专家的）——两者共存、不冲突。
  - highlight **只在边缘描边**（醒目边框 + 轻微外发光），**填充极淡（≈4% 透明度）、绝不覆盖内容**。
- **区域批注工具**（用户紫色矩形）：**紫色边框是识别主体**；填充是**高透明度紫色薄雾**
  （≈5%），透过它能看清下面的内容，不做覆盖。

实现位置：`CanvasWorkspace.jsx` 的 `highlightOverlay`（专家黄/用户紫，fill .04）、
`regionAnnotationOverlay`（紫框 .86 / fill .05）、`pendingChangeOverlay`（fill .04）。

## 4. 回应与可追溯档案（叙事 4）

- 小雷回应一条建议后，原批注贴纸**右上角出现 ✅**；点击后**批注从主画布隐藏**（防过载），但
  **不消失**，收进**左侧对应专家头像下**。
- 点专家头像可见该专家名下每条：**原始批注 / 小雷的修改 / 修改时间 / 专家二次反馈**。→ 整个
  协同过程可追溯，形成**持续演化的研究过程档案**。
- 主要批注回应后，可切**全局(zoom out)视角**：自动显示 Discover 整理结果（已完成 Frame / 已
  回应批注 / 待补材料 / 高频关键词 / 重要洞察 / 专家共同认可的机会）。

## 5. Widget 四类（本 case 用到两类）

Widget 与静态模板不同，带交互。共四类（详见 [widget-examples.md](widget-examples.md)），本 case：

1. **可视化型**（词云 / 柱状图 / 条形图）：**无用户交互，重点是可视化**。读取材料生成图，嵌在
   对应 Frame 内（叙事 2 的词云分析 Widget 嵌在文献研究 Frame）。
2. **轻量输入 + 请求型**：用户轻量输入 → 向 AI 发起请求。典型 UI：**"请求生成" + 选哪位专家 +
   生成几个** → 生成对应张数卡片（叙事 4 的 HMW 生成：请某位专家想 N 个 HMW → N 张卡片）。

（另两类见 widget 文档；本 case 不展开。）

## 6. 范围、命名与免责

- **范围**：Discover → Define（到 HMW 收束）。后续方案设计**暂不做**。
- **专家命名**：使用 TT 设计学院的专家名。演示 / PPT 用小字说明：受技术局限，专家调用为
  **模拟**（skill 展示的是"可以这样调动专家"的机制），并非真实自动调用本人。
