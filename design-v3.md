# Visual Delivery Skill V3 — 产品设计方案

> 设计目标：将 visual-delivery 从「交付物列表」升级为「项目级视觉工作空间」，支持 AI 员工在项目目录下初始化实例，通过状态仪表盘、分层模板体系和项目级反馈池实现透明度、对齐与交付的闭环。

---

## 1. 设计原则

### 1.1 核心原则

- **项目即容器**：每个 `.visual-delivery/` 实例只服务一个项目，全局无需 project_id。
- **行动召唤优先**：主页按「现在该做什么」排序，而非平铺历史汇报。
- **反馈是项目资产**：反馈池归属项目，跨汇报追踪生命周期，最终通过 change_record 体现闭环。
- **模板两层解耦**：结构层（complex-review）决定汇报框架；呈现层（文档/表格/画布/Slides）决定 artifact 的载体。
- **固定布局 + 主题切换**：布局结构固定以保证一致性，仅支持 light/dark 主题与品牌 token 自定义。

### 1.2 视觉基调

- 风格：Clean、professional、focused，参考现有设计规范。
- 信息密度：中高，适合知识工作者快速扫视状态。
- 无障碍：WCAG 2.1 AA，所有状态不止依赖颜色。

保留现有 token 结构，扩展为完整 light/dark 双主题。

### 2.2 Spacing & Radius

- Base unit: `4px`
- 间距阶梯：`4 / 8 / 12 / 16 / 24 / 32 / 48 / 64`
- 卡片内边距：`16px`（紧凑卡片）/ `24px`（页面卡片）
- 圆角：`8px` 默认，`12px` 大卡片，`16px` 弹窗/画板容器

### 2.3 Typography

- 正文字体：系统字体栈（`-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`）
- 代码字体：`'SF Mono', 'Fira Code', monospace`
- 字号阶梯：
  - 标题 H1：`24px / 32px line-height / 600`
  - 标题 H2：`18px / 26px / 600`
  - 标题 H3：`15px / 22px / 600`
  - 正文：`15px / 24px / 400`
  - 辅助文本：`13px / 20px / 400`
  - 标签：`12px / 16px / 500`

### 2.4 主题切换机制

- 在 `<html>` 上设置 `data-theme="light|dark"`。
- 所有组件使用 CSS 变量，禁止硬编码颜色。
- 切换时局部过渡：`transition: background 200ms, color 200ms, border-color 200ms`。
- 默认跟随系统 `prefers-color-scheme`，用户可在设置中显式覆盖。

## 3. 全局导航与布局框架

### 3.1 固定布局结构

采用**三栏固定布局**，结构不随页面变化：

```
┌─────────────────────────────────────────────────────────────┐
│ 顶部导航栏 (Top Bar)                                          │
│ 项目名 · 当前页面标题 · 主题切换 · 设置入口                       │
├──────────┬──────────────────────────────────────┬───────────┤
│          │                                      │           │
│ 左侧导航  │            主内容区                   │ 右侧反馈  │
│ (固定)    │            (动态路由)                 │ 侧边栏    │
│          │                                      │ (可收起)   │
│ 主页      │                                      │           │
│ 日志      │                                      │           │
│ 汇报      │                                      │           │
│ 设置      │                                      │           │
│          │                                      │           │
└──────────┴──────────────────────────────────────┴───────────┘
```

- **左侧导航**：固定宽度 `200px`，包含主页、日志、汇报、设置四个入口。当前项高亮。
- **顶部导航栏**：高度 `56px`，左侧显示项目名称，中间显示当前页面标题，右侧为主题切换、反馈侧边栏开关、设置入口。
- **右侧反馈侧边栏**：默认宽度 `320px`，可收起为 `48px` 的图标条。所有页面共享同一反馈池。
- **主内容区**：填充剩余空间，最大宽度 `1200px` 居中，避免超宽屏拉伸。

### 3.2 路由结构

```
/                      项目主页（状态仪表盘）
/logs                  日志列表与文档
/logs/:logId           单篇日志/文档
/reports               汇报列表
/reports/new           新建汇报（选择模板/结构）
/reports/:reportId     单篇汇报详情
/settings              设置（项目、主题、host、密钥）
```

### 3.3 反馈侧边栏全局行为

- 右侧边栏是**项目级反馈池**的视图，不随单个 report 关闭而清空。
- 在任何页面选中文本/元素，均可触发「添加反馈」浮动按钮。
- 侧边栏内按状态分组：待处理、已处理、已归档。
- 每条反馈显示：提出时间、来源（哪个 report/log）、当前处理 report（如有）、change_record 摘要。

## 4. 项目主页：状态仪表盘

### 4.1 设计目标

项目主页不是汇报列表，而是**项目当前状态的仪表盘 + 行动召唤入口**。用户打开后能在 3 秒内知道：

1. 项目整体健康度/阶段
2. 现在最该关注什么（待处理反馈、进行中的汇报、阻塞的对齐）
3. 快速进入下一个动作

### 4.2 页面结构（自上而下）

#### 4.2.1 顶部项目卡（Project Header Card）

```
┌────────────────────────────────────────────────────────────┐
│  [项目图标]  项目名称                          [阶段标签]   │
│  一句话项目描述 / 最近目标                                      │
│  [继续当前工作]  [新建汇报]  [查看待处理反馈]                    │
└────────────────────────────────────────────────────────────┘
```

- 项目图标：首字母或自定义 logo，圆形 `48px`。
- 阶段标签：如「需求对齐中」「执行中」「评审中」「已收尾」，颜色语义化。
- 三个主按钮按优先级排序：
  1. **继续当前工作**：跳转到最后未完成的 report 或 alignment。
  2. **新建汇报**：打开模板选择器。
  3. **查看待处理反馈**：打开右侧反馈侧边栏并过滤到待处理。

#### 4.2.2 状态指标行（Status Metrics Row）

4 个指标卡片横向排列（响应式：移动端 2×2）：

| 指标 | 含义 | 阈值告警 |
|------|------|----------|
| 待处理反馈 | 项目反馈池中 handled=false 的数量 | >3 时标橙，>5 时标红 |
| 进行中的汇报 | status=draft/progress 的 report 数 | >1 时提示合并 |
| 最近 7 天日志 | 最近一周的 log 条目数 | 0 时提示记录 |
| 项目活跃度 | 最近 30 天 report/log 总数 | 用于长期感知 |

#### 4.2.3 行动召唤区（Primary Actions）

按紧急程度排序的三张行动卡片：

1. **阻塞对齐待确认**（alignment 状态）
   - 仅当存在 active alignment 时显示。
   - 红色边框，顶部固定提示条。
   - 卡片显示：对齐标题、阻塞的问题、「去确认」按钮。

2. **待处理反馈**（Feedback To Address）
   - 列出最近 3 条待处理反馈。
   - 每条显示：摘要、来源 report、提出时间、「在下次汇报处理」按钮。
   - 点击后打开 report 创建流程，并自动关联这些反馈。

3. **继续当前汇报**（Continue Report）
   - 显示最近一份未完成的 report 标题、模板类型、进度百分比。
   - 进度基于 report 内 section 完成度或 agent 自评。

#### 4.2.4 最近动态流（Recent Activity Feed）

时间线形式展示最近 7 条项目事件：

- report 创建/提交
- feedback 提出/处理/确认
- alignment 创建/解决
- log 自动生成

每条事件包含：图标、标题、时间、关联对象链接。

#### 4.2.5 快捷入口网格（Quick Access Grid）

底部 4 个固定快捷入口：

- 所有汇报
- 所有日志/文档
- 反馈池（全量）
- 项目设置

### 4.3 空状态

- 新项目首次初始化后，主页显示欢迎引导：
  - 项目信息已配置 ✓
  - 创建第一份汇报 → 引导选择模板
  - 查看示例（可选）

### 4.4 响应式行为

- **Desktop（≥1024px）**：完整三栏，指标 4 列，行动卡片 3 列。
- **Tablet（768px–1023px）**：左侧导航可收起，指标 2×2，行动卡片单列堆叠。
- **Mobile（<768px）**：左侧导航变为底部 tab 栏，右侧反馈侧边栏变为底部抽屉，内容单列。

## 5. 模板体系：两层分类

### 5.1 分类逻辑

| 层级 | 模板 | 作用 | 决定因素 |
|------|------|------|----------|
| **结构层** | complex-review | 定义汇报的骨架：多 section、artifact + 叙事 + 反馈循环 | 任务复杂度、是否需要长期跟踪 |
| **呈现层** | document / table / canvas / slides | 定义每个 section 中 artifact 的渲染形态 | 内容类型、最适合消费的媒介 |

**关键规则**：

- 结构层和呈现层不是二选一，而是**组合使用**。
- 一个 complex-review 汇报中，每个 section 可以嵌入不同呈现层模板。
- 普通短周期汇报可跳过 complex-review，直接用一个呈现层模板作为单 section 汇报。
- Agent 自动判断，但判断逻辑对用户透明（在 report 创建前显示「预计使用模板」）。

### 5.2 结构层：complex-review

#### 5.2.1 适用场景

- 长期任务：论文、行业研究、软件开发里程碑、复杂提案。
- 需要多轮反馈和修改跟踪。
- 包含多个 section，每个 section 有独立的结论/证据/artifact。

#### 5.2.2 页面结构

```
┌─────────────────────────────────────────────────────────────┐
│  汇报标题                                                    │
│  [标签：complex-review]  [状态]  [创建时间]                    │
├─────────────────────────────────────────────────────────────┤
│  Section 导航（左侧垂直）                                      │
│  · 1. 背景与目标                                              │
│  · 2. 关键发现                                                │
│  · 3. 方案对比                                                │
│  · 4. 下一步行动                                              │
├─────────────────────────────────────────────────────────────┤
│  当前 Section 内容区                                          │
│  ─────────────────────────────────────────────────────────  │
│  Section 标题 + 叙事文本                                       │
│  [Artifact：document / table / canvas / slides]               │
│  ─────────────────────────────────────────────────────────  │
│  [← 上一 section]              [下一 section / 提交汇报 →]      │
└─────────────────────────────────────────────────────────────┘
```

- Section 导航固定左侧，显示完成状态（未开始/进行中/已完成）。
- 每个 section 由「叙事文本 + artifact」组成。
- Artifact 根据内容类型选择呈现层模板渲染。
- 底部 section 切换按钮始终可见。

#### 5.2.3 数据模型扩展

```json
{
  "structure": "complex-review",
  "sections": [
    {
      "id": "sec-1",
      "title": "背景与目标",
      "narrative": "...",
      "artifact": {
        "type": "document",
        "content": { "type": "ui_spec", "ui_spec": { ... } }
      },
      "status": "completed"
    }
  ]
}
```

### 5.3 呈现层模板

#### 5.3.1 文档模板（document）

- 形态：交互式文本文档，类似 Notion/Google Docs 的轻量版。
- 交互：
  - 段落可折叠（长文档）。
  - 标题自动生成目录。
  - 文本选中后弹出「添加反馈」按钮。
  - 支持内嵌表格、代码块、图片。
- 适用：提案、报告、研究结论、说明性内容。

#### 5.3.2 表格模板（table）

- 形态：交互式表格，支持排序、过滤、展开详情。
- 交互：
  - 表头点击排序。
  - 行悬停高亮，点击展开详情面板。
  - 单元格可批注（选中后添加反馈）。
  - 支持导出 CSV。
- 适用：对比分析、指标看板、数据汇总、方案评分表。

#### 5.3.3 画布模板（canvas） ← 参考 cowart

- 形态：基于 tldraw 思路的无限画布，支持自由排布。
- 元素类型：
  - 图片 holder（可上传或 AI 生成填充）
  - 便签（sticky note）
  - 箭头/连线（用于标注关系）
  - 手绘图形
  - 文本块
- AI 协作：
  - Agent 可读取画布当前选择状态和元素列表。
  - 用户可以用箭头+文字标注「改这里」，Agent 据此修改。
  - 画布状态可导出为快照，嵌入 report。
- 反馈循环：
  - 用户在画布上添加批注 → 进入反馈池。
  - Agent 处理后在同一画布上更新元素 → 自动生成 change_record。
- 适用：视觉方案评审、架构图、情绪板、设计迭代、概念脑图。

#### 5.3.4 Slides 模板（slides）

- 形态：幻灯片播放器 + 缩略图导航。
- 交互：
  - 左侧缩略图，右侧主视图。
  - 每页支持文本、图表、图片。
  - 演讲者备注（仅创建者可见）。
  - 整页或局部可添加反馈。
- 适用：需要逐页讲解的方案、路演、阶段性汇报。

### 5.4 模板选择器（新建汇报）

新建汇报时，先由 agent 推荐，用户可覆盖。选择器按两层分类展示：

```
┌─────────────────────────────────────────────────────────────┐
│  新建汇报                                                    │
│  Agent 推荐：complex-review + document                        │
├─────────────────────────────────────────────────────────────┤
│  1. 选择结构                                                 │
│  [○ 标准汇报]  [● 复杂任务评审]                               │
│                                                             │
│  2. 选择主要呈现形式（complex-review 下每个 section 可换）      │
│  [文档]  [表格]  [画布]  [Slides]                             │
│                                                             │
│  3. 关联待处理反馈（可选）                                     │
│  [ ] 反馈 A：调整目标用户定义                                  │
│  [✓] 反馈 B：补充竞品对比                                     │
├─────────────────────────────────────────────────────────────┤
│  [取消]                              [创建汇报草稿]           │
└─────────────────────────────────────────────────────────────┘
```

- 标准汇报：单 section，直接选呈现层模板。
- complex-review：多 section，先确定整体结构，再为第一个 section 选呈现层。
- 关联反馈：自动列出项目反馈池中待处理的反馈，用户可勾选让本次汇报处理。

## 6. 画布模板详细设计（参考 cowart）

### 6.1 核心借鉴点

Cowart 的核心价值：

1. **可视化上下文**：用户可以在画布上圈、箭头、文字标注，AI 能看到这些空间化指令。
2. **指哪改哪**：反馈不再是抽象文字，而是绑定到具体视觉元素上的批注。
3. **迭代快照**：每次修改后可以保存画布快照，对比前后差异。

### 6.2 画布工具栏

顶部固定工具栏（类似 Figma/tldraw）：

| 工具 | 功能 |
|------|------|
| 选择 (V) | 选择、移动、缩放画布元素 |
| 便签 (N) | 添加彩色便签 |
| 文本 (T) | 添加文本块 |
| 箭头/连线 (A) | 绘制带箭头的连线，用于标注 |
| 手绘 (P) | 自由手绘 |
| 图片 (I) | 上传图片或创建 AI 图片 holder |
| 帧/画框 (F) | 创建容器帧，用于组织内容 |

### 6.3 AI 图片 Holder

参考 cowart 的关键元素：

- 用户在画布上拖拽创建一个「AI 图片占位框」。
- 占位框显示提示词输入框。
- Agent 根据提示词和画布上下文生成图片并填入。
- 生成后可继续用箭头标注修改意见，Agent 据此迭代。

### 6.4 批注与反馈

- 任何元素或区域可被「添加批注」，批注气泡显示在元素旁。
- 批注自动同步到项目反馈池，标记来源为「画布：reportId」。
- 处理反馈时，Agent 在画布上修改对应元素，并在批注下追加 change_record。

### 6.5 画布快照与版本

- 每次 report 提交时，画布自动保存为只读快照。
- 用户可在反馈池中看到「修改前快照」与「修改后快照」的对比。
- 快照以 PNG/SVG 形式导出，也可作为 report 的 embedded asset。

### 6.6 画布 UI Spec 结构

```json
{
  "type": "canvas",
  "canvas": {
    "version": "1.0",
    "elements": [
      { "id": "img-1", "type": "image", "x": 100, "y": 100, "src": "...", "caption": "..." },
      { "id": "note-1", "type": "sticky", "x": 300, "y": 150, "text": "...", "color": "yellow" },
      { "id": "arrow-1", "type": "arrow", "from": "note-1", "to": "img-1", "label": "调整这里" }
    ],
    "viewport": { "x": 0, "y": 0, "zoom": 1.0 }
  }
}
```

## 7. 初始化流程设计

### 7.1 触发条件

首次在项目目录下运行 `node scripts/start.js` 时，检测到 `.visual-delivery/` 不存在，进入初始化流程。

### 7.2 初始化步骤

#### Step 1: 项目信息

```
┌─────────────────────────────────────────────────────────────┐
│  初始化 visual-delivery                                      │
│  为你的项目创建一个视觉工作空间                                │
├─────────────────────────────────────────────────────────────┤
│  项目名称 *                                                  │
│  [________________________]                                  │
│  项目描述（一句话）                                           │
│  [________________________]                                  │
│  项目阶段                                                    │
```

## 8. 反馈系统与 change_record

### 8.1 反馈生命周期

```
提出 (proposed) → 追踪 (tracked) → 处理 (addressed) → 体现变更 (confirmed)
```

- **proposed**：用户在 report/log/画布上选中内容并提交反馈。
- **tracked**：反馈进入项目反馈池，等待处理。
- **addressed**：某份 report 明确关联并处理了该反馈。
- **confirmed**：用户确认 change_record，反馈归档。

### 8.2 Feedback 数据模型

```json
{
  "id": "fb-1",
  "status": "addressed",
  "source": {
    "type": "report",
    "reportId": "r-1",
    "sectionId": "sec-2",
    "elementId": "el-1",
    "selectedText": "目标用户定义"
  },
  "content": "目标用户定义需要更聚焦",
  "author": "user",
  "createdAt": "2026-06-29T08:00:00Z",
  "changeRecord": {
    "reportId": "r-2",
    "summary": "在 r-2 中重新划分了目标用户为...",
    "diff": "...",
    "confirmedAt": "2026-06-29T10:00:00Z"
  }
}
```

### 8.3 change_record 展示

当一条 feedback 被处理后，在反馈侧边栏和 report 中显示：

```
┌─────────────────────────────────────────┐
│  反馈：目标用户定义需要更聚焦            │
│  来源：r-1 > 关键发现                    │
├─────────────────────────────────────────┤
│  处理记录                                │
│  在 r-2 中已调整：                       │
│  · 原定义：所有职场人群                  │
│  · 新定义：25-35 岁一线城市知识工作者     │
│  [查看 r-2 相关 section]                 │
├─────────────────────────────────────────┤
│  [确认已解决]  [仍有问题，继续反馈]        │
└─────────────────────────────────────────┘
```

## 11. 关键交互流程

### 11.1 新建汇报流程

```
用户点击「新建汇报」
  ↓
Agent 推荐模板（结构层 + 呈现层）
  ↓
用户确认或修改模板，勾选待处理反馈
  ↓
创建 report 草稿，自动生成 log 条目
  ↓
用户进入 report 编辑/查看页面
  ↓
用户提交 report，状态变为 submitted
  ↓
自动生成 log，更新主页状态
```

### 11.2 反馈处理流程

```
用户在 report/log/画布上选中内容 → 添加反馈
  ↓
反馈进入项目反馈池（status=tracked）
  ↓
用户在主页/反馈池看到待处理反馈
  ↓
用户选择「在下次汇报处理」或 Agent 主动关联
  ↓
新建 report 时勾选待处理反馈
  ↓
Agent 在 report 中处理反馈，生成 change_record
  ↓
用户确认 change_record → feedback status=confirmed
  ↓
反馈归档，主页指标更新
```

### 11.3 画布协作流程

```
Agent 创建 canvas report
  ↓
用户查看画布，添加图片/便签/箭头批注
  ↓
用户提交批注 → 进入反馈池
  ↓
Agent 读取画布选择状态和批注
  ↓
Agent 修改画布元素（换图、调位置、加文字）
  ↓
保存画布快照，生成 change_record
  ↓
用户对比快照，确认或继续批注
```
