# Visual Delivery Skill — V4 产品设计文档

> 本文档基于原始产品需求规格与孙效华教授交互设计专家视角的优化讨论，记录 V4 方案的核心设计决策与架构方向。
> 
> 讨论时间：2026-06-29 | 版本：V4 草案

---

## 一、产品定位与核心问题

### 场景定义

**AI 员工模式**：用户是领导，负责下指令和描述需求。AI 负责对齐领导需求和执行任务。这种模式下，agent 的能力不仅要连续准确地完成用户指定的任务，也需要学会和老板沟通对齐。

我们关注在 AI 员工模式下，用户使用 agent 执行**项目**（而非一次性简单任务）的情境。

**典型场景**：撰写提案/论文/报告、软件开发、科学研究/行业研究

### 目标用户

日常用户（非程序员）

### 三大核心问题

| 问题 | 现状痛点 | Skill 解决方式 |
|------|---------|---------------|
| **工作透明度** | 用户不清楚 AI 员工做了什么 | 日志 + 工作文档自动记录 |
| **对齐** | 如何像真实员工一样和相关方对齐需求 | 汇报页面 + 结构化反馈交互 |
| **交付评审** | 智能体如何优雅地向领导汇报 | 产出 + 叙事结构的评审页面 |

### Skill 定位

支持 AI 员工通过**视觉界面**来对齐、评审、记录。核心在于 skill 让 agent 能够搭建一个视觉界面平台，支持以上三大问题的解决。

## 二、信息架构重构——从"交付物"到"项目容器"

> 这是整个 V4 方案的**地基级决策**，所有后续设计建立在此之上。

### 现状（V2/V3）：Delivery-Centric

- `POST /api/deliveries` 创建独立交付物
- Dashboard 平铺展示所有 delivery
- 每个 delivery 之间没有逻辑关联
- 反馈锁在单个 delivery 内部：`feedback.json` 存在 `data/deliveries/{id}/`
- 生命周期：draft → commit → resolve（"评审→打分"逻辑）

### V4 方案：Project-Centric

- **Project 成为顶层容器**，delivery/report 降级为项目下的一个事件
- `.visual-delivery/` 目录就是项目本身的数据空间
- 不需要 `project_id` 字段区分——整个 server 实例为一个项目服务
- 视觉网站实例**跟随项目**：skill 在项目目录下初始化

### 架构简化效应

Project 级实例化带来三个直接简化：

1. **反馈归属天然清晰** — 所有反馈都在同一数据空间，天然 project-scoped
2. **Server 生命周期绑定项目** — 项目初始化时启动，项目结束时可归档
3. **数据模型少一层间接** — 不需要多租户路由逻辑

### 约束

一个端口同时只能服务一个项目。需后续处理端口分配或动态端口。

## 三、汇报-日志-反馈三角关系

三者不是并列的，而是**以反馈池为中心的循环**：

```
Report 创建 → 触发 Log 条目 → 用户在 Report 中提 Feedback 
→ Feedback 进入项目级反馈池 → 后续 Report 处理反馈并体现变更
```

### 反馈数据模型演进

**现有模型**（delivery-centric）：

```json
{
  "id": "f_...",
  "delivery_id": "d_...",
  "kind": "annotation|interactive",
  "payload": {...},
  "handled": false
}
```

**V4 模型**（project-centric + 反馈循环）：

```json
{
  "id": "f_...",
  "report_id": "r_...",          // 在哪份汇报中提出
  "kind": "annotation|interactive",
  "payload": {...},
  "target": {...},
  "status": "pending|resolved",  // 替代 handled（更语义化）
  "change_record": {
    "addressed_in_report_id": "r_...",   // 在哪份后续汇报中被处理
    "change_summary": "根据反馈调整第三章结构",
    "diff_refs": ["component-3:line-12"]
  },
  "created_at": "...",
  "resolved_at": "...",
  "resolved_by": "agent"
}
```

**关键变化**：
- `status` 替代 `handled`（更语义化）
- `report_id` 指明出处，但反馈归属项目
- `change_record` 支持反馈循环——改了什么、在哪里改的

### 反馈生命周期：四段式

从 V2/V3 的三段式 `draft → commit → resolve`（"评审→打分"逻辑）进化为：

```
提出(tracked) → 追踪(在反馈池中) → 体现(addressed + change_record) → 瑞认(confirmed)
```

多了"体现"这一环——**反馈不是对某次交付的终评，而是持续协作对话中的一个回合**。

## 四、汇报的"产出 + 叙事"双重身份

### 核心洞察

AI 员工的"汇报"汇报的不仅是产出本身，还有**决策逻辑和取舍**。真实员工做评审汇报，不是把 PPT 往桌上一扔，而是讲清楚：我做了什么、为什么这么做、哪里需要拍板、哪些地方我拿不准。

### 叙事结构设计

复杂任务评审模板的核心不是把 artifact 嵌进去（那只是 iframe 或截图的事），而是设计一种**叙事结构**——每个 section 是一个"叙事单元"：

```
Section = Artifact 呈现 + 决策说明 + 反馈触发点
```

现有 RuntimeRenderer 渲染的是扁平的 components 列表（markdown、data_view、decision_form、metric_grid）。V4 的评审模板需要一种 **section 抽象**。

## 五、反馈循环的可追溯性

> 这是整个方案里**设计难度最高**的部分。

### 变更感知问题

要让用户在新版汇报页面中，**不用翻历史记录**，就能知道"我上次提的反馈被怎么处理了"。

### 解决方案：变更摘要条（Change Log Strip）

每条反馈在 resolve 时，关联一个 **change record**。在下一版汇报页面上，用一个变更摘要条展示：

```
"根据你的 3 条反馈，我做了以下调整：…"
每条变更可点击跳转到对应的内容位置
```

用户看到的闭环：**我提了反馈 → agent 改了 → 改了什么我一眼能看到**

## 六、模板选择——两层决策模型

### 问题

五种模板如果让用户选，用户不知道选什么。如果让 agent 每次都问，是把决策负担甩给领导。

### 决策：Agent 自动路由 + 用户可否决

模板选择由 agent 根据任务类型自动判断，但判断逻辑对用户透明。就像好员工不会问领导"我该用 PPT 还是 Word"，而是自己判断后说"这个数据比较多，我用表格形式给你看"。

### 两层分类模型

| 层级 | 触发条件 | 选项 |
|------|---------|------|
| **结构层** | 任务是否 long-horizon、多阶段、需要评审 | 复杂任务评审 vs 单次汇报 |
| **呈现层** | 产出类型、数据量、反馈复杂度 | 文本文档 / 表格 / 画布 / Slides |

**关键区别**：结构层回答"这次汇报需要多少结构化决策"，呈现层回答"我用什么形式呈现内容"。一个复杂任务评审可能内部使用 slides 或文档作为 artifact 呈现形式。

### Agent 路由逻辑

```
1. 先判断是否需要"复杂任务评审"结构（看任务是否是 long-horizon、多阶段、需要评审）
2. 再判断内容呈现形式（slides / 文档 / 表格 / 画布）
```

在 SKILL.md Step 3 前面加一步**模板路由**，告知用户选择理由，用户可以否决。

## 七、功能模块详细设计

### 7.1 记忆和数据

- 在项目下面生成 `.visual-delivery/` 临时目录，存放 skill 产生的数据
- 包含界面代码、项目相关数据（文档等）、个性化配置
- 数据目录结构（V4 规划）：

```
.visual-delivery/
  server/           — Express 后端
  ui/               — 前端代码 + dist
  data/
    reports/        — 汇报数据
      {reportId}/
        report.json
        feedback.json
    logs/           — 日志条目
    documents/      — 工作文档
    settings.json   — 项目配置
  design/           — 设计规范
```

### 7.2 网站 Hosting

- 本地启动（localhost:3847）
- 局域网启动（可以把链接发给别人）
- 第三方在线平台 host 服务（适用于远程使用）
- 支持密钥访问

### 7.3 平台模板（初始界面）

Skill 提供初始化网站界面，包含：
- **项目主页** — 项目状态仪表盘（不是汇报列表）
- **导航**：日志（含文档）、汇报、设置

### 7.4 项目主页设计

主页回答三个问题：
1. **项目进展到哪了？** — 当前进行中的汇报 / 待处理反馈数
2. **最近发生了什么？** — 日志时间线摘要
3. **接下来需要我做什么？** — 突出待反馈或待决策的汇报

**核心原则**：按**行动召唤**排序，而非按时间平铺。

### 7.5 个性化

- 布局不变（三栏：Sidebar + TopBar + Content）
- 支持 light/dark 主题切换
- 功能维持不变
- 用户可在 skill 约束下定制视觉样式

### 7.6 汇报页面模板

以下模板不是展示方向或示例，而是 V4 汇报系统需要实现的基础模板能力。Agent
可以根据任务自动路由到其中一种，也可以在 `complex-review` 中混合使用多种
呈现层模板。

#### A 类：内容呈现型模板（agent 根据内容类型自动选择）

| 模板 | 适用场景 | 核心交互 |
|------|---------|---------|
| 交互式文本文档 | 分析、论证、说明为主的汇报 | 文档形式 + 符合汇报需求的交互式组件 |
| 交互式表格 | 数据为主的汇报 | 表格为主要汇报形式 + 数据反馈交互 |
| 交互式画布 | 设计创意、头脑风暴、灵感采集、产品设计思路推进等视觉协作场景 | 参考 Cowart 类画布式协作体验，基于 tldraw 实现无限画布、批注和添加信息 |
| 交互式 Slides | 复杂、多章节、结构化叙事 | 多 slide 交互式汇报 |

#### B 类：任务评审型模板（由任务特征触发）

| 模板 | 适用场景 | 核心特性 |
|------|---------|---------|
| 复杂任务评审 | Long-horizon task | 1. 多 section 划分 2. 多种交互式组件 3. Artifact + 汇报叙事双重呈现 |

如果用户有特殊需求，按意图创建新模板（可基于现有模板）。

#### 模板实现要求

| 模板 | 必须实现的基础能力 | 暂不进入第一版的高级能力 |
|------|-------------------|-------------------------|
| 文本文档 | Markdown / HTML 渲染、目录导航、段落级反馈、决策点组件 | 完整文档编辑器 |
| 表格 | 数据表格、筛选/排序、行级反馈、字段级说明 | 类 BI 的复杂透视分析 |
| 画布 | tldraw 无限画布、文本/图片/卡片节点、批注节点、选区反馈、画布状态保存 | 多人实时协作、复杂矢量设计工具 |
| Slides | slide 导航、逐页汇报、页级反馈、重点决策页 | 完整演示文稿编辑器 |
| Complex-review | section schema、artifact + 叙事 + 反馈触发点、多模板混排 | 复杂项目管理和里程碑系统 |

画布模式的核心不是“把一张图放进页面”，也不是一次性静态汇报截图，而是为设计创意、
头脑风暴、灵感采集、产品设计思路推进等任务提供一个 agent 与用户共同工作的项目级
无限画布。Agent 可以在画布中持续添加内容、整理线索、放置方案、生成设计说明或推进
产品思路；用户也可以在同一画布上批注、补充素材、圈选区域并反馈。

画布模板的产品参考方向是 Cowart（https://github.com/zhongerxin/cowart）类体验：
在当前项目旁边打开一个本地 tldraw 无限画布，
让 Codex/agent 和用户在同一个视觉空间中协作，而不是把画布当作普通报告组件。Visual
Delivery 负责把 agent 生成的产出、说明、待决策点和用户反馈映射为画布节点与项目级
反馈记录。画布状态应归属当前项目工作空间，后续汇报可以引用画布页面、节点、选区或
截图作为 artifact。

第一版不追求完整设计工具能力，但必须支持视觉协作所需的查看、批注、补充信息、选区
反馈、状态保存和后续汇报引用。新建画布保持空白，避免出现与当前任务无关的预制区域；
需要区分 agent 内容、用户补充和共享决策时，应由具体脚手架按任务场景生成清晰 section，
避免所有素材堆叠在同一层级。

### 7.7 反馈交互

- **右侧统一侧边栏**：所有用户提交的反馈呈现在侧边栏
- 侧边栏支持直接输入文字反馈
- 反馈条目支持删除和修改
- 本地保存
- **全局反馈**：页面上的文本和图片等媒体支持选中后弹出窗口提交反馈
- **反馈循环可视化**：变更摘要条展示"根据反馈做了什么调整"

### 7.8 反馈支持

- 所有反馈存储（待处理状态）
- 智能体沟通时调用反馈标注信息
- 处理完成后存档（已处理）
- 反馈循环：页面上直观体现根据用户反馈做的更改

## 八、使用流程

### 初始化

用户在项目中初始化 skill → skill 调用模板核对项目信息、模式选择、视觉样式 → 生成项目基本页面

### 汇报

用户让智能体汇报（或智能体自动识别）→ 智能体根据模板路由逻辑选择汇报形式 → 生成汇报页面 → 用户 review + 直接反馈

### 记录

每次汇报自动撰写日志 → 如果用户在项目中记录工作文档则提供链接 → 如果没有记录则同时撰写工作文档 → 文档可在视觉界面平台中查看和标注

### 反馈

用户在平台上提交反馈 → 智能体接到指令后读取反馈 → 添加到对话上下文 → 支持需求对齐

## 九、与现有架构的对照

| 维度 | V2/V3（现状） | V4（目标） |
|------|-------------|-----------|
| 信息架构 | Delivery-centric（交付物平铺） | Project-centric（项目容器） |
| 反馈归属 | 绑定在 delivery 上 | 项目级反馈池（跨 report 追踪） |
| 反馈生命周期 | 三段式：draft→commit→resolve | 四段式：提出→追踪→体现→确认 |
| 主页 | Dashboard（delivery 列表） | 项目状态仪表盘（行动召唤排序） |
| 模板选择 | 无路由，agent 自行决定 | 两层路由（结构层 + 呈现层） |
| 汇报内容 | 产出呈现 | 产出 + 叙事结构 |
| 反馈循环 | 标记 handled=true | change_record + 变更摘要条 |
| 数据存储 | deliveries/{id}/feedback.json | reports/{id}/feedback.json + 项目级反馈池 |

## 十、待确认与下一步

### 已确认的设计决策

1. ✅ Project 作为顶层容器，反馈挂到 project 层级
2. ✅ 不支持项目切换，skill 实例跟随项目
3. ✅ 项目主页是"状态仪表盘"而非"汇报列表"
4. ✅ 模板两层分类（结构层 + 呈现层）
5. ✅ 反馈生命周期四段式（提出→追踪→体现→确认）
6. ✅ 五类汇报模板都需要实现基础版本：文档、表格、画布、Slides、Complex-review
7. ✅ 画布模板参考 Cowart 类体验，使用 tldraw 实现
8. ✅ 画布模式适用于设计创意、头脑风暴、灵感采集、产品设计思路推进等协作场景

### 待细化

- 汇报模板的 section 抽象具体 schema
- 各汇报模板的数据 schema 与渲染协议
- 变更摘要条（Change Log Strip）的 UI 设计细节
- Agent 模板路由的判断规则表
- 日志与工作文档的关联机制
- 端口分配策略（多项目并发）
- tldraw 画布模板的节点类型、反馈映射、状态保存格式，以及与汇报/反馈池的引用关系
- 密钥访问机制
